# Ad Accounts Manager

Веб-приложение для учёта рекламных кабинетов — тестовое задание.
Имитация системы учёта: добавление кабинетов, таблица с фильтрами и поиском,
ручная смена статусов, дашборд и страница возвратов с экспортом в CSV.
Реальные рекламные API не используются — данные хранятся в собственной БД.

Дизайн и палитра ориентированы на фирменный стиль
[lidera-agency.com](https://lidera-agency.com): тёмная тема, акценты
magenta `#FF058C` / teal `#0ABAB5`, шрифт Montserrat.

## Демо

🔗 **Живая версия: http://78.17.155.32/** — открывается по ссылке, авторизация не нужна.

Хостится в Docker на сервере (Postgres + Django + nginx). Разделы ниже нужны,
только если хочется запустить или пересобрать проект самому — для проверки кода.

## Стек

| Слой      | Технологии                                                    |
| --------- | ------------------------------------------------------------- |
| Backend   | Python 3.11, Django 5, Django REST Framework, django-filter   |
| БД        | SQLite (локально) / PostgreSQL (прод, через `DATABASE_URL`)   |
| Frontend  | Vue 3, TypeScript, Vite, Pinia, Vue Router                    |

Монорепозиторий: `backend/` — REST API, `frontend/` — SPA.

## Функционал

1. **Добавление кабинетов** — форма с полями ID, Тип (Agency / BM / Personal),
   Селлер и Отдел. Можно добавлять несколько подряд.
2. **Таблица кабинетов** — колонки ID, тип, селлер, отдел, статус, дата;
   фильтры по отделу и статусу, поиск по ID.
3. **Смена статусов вручную** — клик по статусу переключает его по циклу
   `Выдан → В работе → Бан → Возврат`; через меню можно выставить любой статус.
4. **Дашборд** — всего кабинетов, живых, в бане, возвраты; разбивка по отделам.
5. **Возвраты** — отдельная страница со статусами «Бан» и «Возврат» и кнопкой
   экспорта в CSV.

При старте база предзаполняется 16 случайными кабинетами (командой `seed`),
как минимум по два на каждый статус — чтобы всё было видно сразу.

## Локальный запуск

> Живое приложение уже доступно по ссылке выше — этот раздел нужен для запуска
> и ревью кода локально. Нужны два терминала — для бэкенда и фронтенда.

### 1. Backend (порт 8000)

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python manage.py migrate
python manage.py seed --fresh      # предзаполнить демо-данными
python manage.py runserver
```

API поднимется на `http://127.0.0.1:8000/api/`.

### 2. Frontend (порт 5173)

```bash
cd frontend
npm install
npm run dev
```

Приложение откроется на `http://localhost:5173`. В режиме разработки запросы
`/api/*` проксируются на бэкенд (см. `frontend/vite.config.ts`), поэтому
дополнительная настройка не нужна.

## Переменные окружения

Значения по умолчанию рассчитаны на локальный запуск; для продакшена см.
`backend/.env.example` и `frontend/.env.example`.

**Backend** (`backend/.env`):

| Переменная             | Назначение                                        |
| ---------------------- | ------------------------------------------------- |
| `SECRET_KEY`           | Секретный ключ Django                             |
| `DEBUG`                | `True` локально, `False` в проде                  |
| `ALLOWED_HOSTS`        | Разрешённые хосты через запятую                   |
| `DATABASE_URL`         | Строка Postgres; без неё используется SQLite      |
| `CORS_ALLOWED_ORIGINS` | Origin фронтенда                                  |

**Frontend** (`frontend/.env`):

| Переменная      | Назначение                                              |
| --------------- | ------------------------------------------------------- |
| `VITE_API_BASE` | Origin бэкенда в проде; пусто локально (работает прокси) |

## API

| Метод  | Эндпоинт                      | Описание                              |
| ------ | ----------------------------- | ------------------------------------- |
| GET    | `/api/accounts/`              | Список (фильтры `department`, `status`, `search`, `ordering`) |
| POST   | `/api/accounts/`              | Создать кабинет                       |
| PATCH  | `/api/accounts/{id}/`         | Обновить (в т.ч. статус)              |
| POST   | `/api/accounts/{id}/advance/` | Следующий статус по циклу             |
| DELETE | `/api/accounts/{id}/`         | Удалить                               |
| GET    | `/api/accounts/stats/`        | Сводка для дашборда                   |
| GET    | `/api/accounts/export/`       | CSV по текущим фильтрам               |
| GET    | `/api/accounts/meta/`         | Списки типов, отделов, статусов, селлеров |

## Деплой (Docker)

В репозитории есть готовый прод-стек: Postgres + Django (gunicorn) + nginx,
отдающий собранный SPA и проксирующий `/api` на бэкенд. Наружу публикуется
только nginx.

```bash
git clone https://github.com/Makarov1e/ad-accounts-manager.git
cd ad-accounts-manager
cp .env.prod.example .env      # и заполнить SECRET_KEY, POSTGRES_PASSWORD, ALLOWED_HOSTS
docker compose up -d --build   # соберёт образы и поднимет стек
```

Миграции и предзаполнение демо-данными (`seed --if-empty`) выполняются
автоматически при старте контейнера. Приложение будет доступно на `http://<host>/`
(порт настраивается через `WEB_PORT` в `.env`). SPA и API работают на одном
origin, поэтому CORS не задействован.
